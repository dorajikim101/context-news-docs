# 컨텍스트 뉴스 MVP 1차안 v0.1

## 1. 이 문서의 목적
이 문서는 최종 제품 비전 문서를 대체하지 않는다.

최종적으로 지향하는 것은:
- 서사의 생성·성장·분기·합류·쇠퇴를 추적하는 컨텍스트 인텔리전스 서비스
- 공용 메인 서사 그래프 + 개인화 서사 그래프
- 정치/경제/산업/리서치/proxy 층을 아우르는 서사 구조화 시스템

그러나 그 비전을 **처음부터 그대로 구현하면 개발/운영 복잡도가 너무 높다.**

따라서 이 문서는 다음을 정의한다.

> 최종 목표는 유지하되,
> **MVP 단계에서는 무엇을 버리고 무엇만 만들어야 하는가**

즉 이 문서는 **컨텍스트 뉴스의 1차 실행안**,
정확히는 **반자동 서사 편집 CMS 중심의 MVP 설계안**이다.

---

## 2. 전제: 원래 목표와 MVP 목표는 다르다

### 원래 목표
원래 시도하려는 것은 다음과 같다.
- 서사를 중심 단위로 한 뉴스 해석 시스템
- 메인 서사 그래프와 개인 서사 그래프
- 다양한 데이터 층(정치/경제/산업/리서치/proxy/대중)을 반영한 구조
- 행동/자금/기관 움직임까지 반영한 판세 해석
- 장기적으로는 고도화된 시각화와 개인화까지 포함

### MVP 목표
MVP에서 증명해야 하는 것은 더 작다.

핵심 질문은 이것이다.

> 좁은 도메인(초기에는 크립토)에서,
> 제한된 소스와 인간 편집을 기반으로,
> 서사 / 주장 / 근거 / 반론 / 상태를 구조화했을 때,
> 실제로 기존 뉴스 소비보다 더 나은 맥락 해석 경험을 줄 수 있는가?

즉 MVP는 **최종 서비스 축소판**이 아니라,
최종 비전을 검증하기 위한 **현실적인 편집 시스템**이어야 한다.

---

## 3. MVP의 핵심 재정의
MVP의 본체는 외부 사용자용 뉴스 앱이 아니다.

MVP의 본체는:
**반자동 서사 편집 CMS + 해석 시스템**
이다.

왜 이렇게 보느냐:
- 서사의 품질은 자동화보다 편집 품질에 좌우된다.
- 초기에 가장 어려운 건 UI보다 데이터 정합성이다.
- claim extraction, counterpoint 연결, narrative split/merge 판정은 사람 검수 없이는 신뢰가 무너지기 쉽다.
- 따라서 초기에 먼저 필요한 것은 사용자용 화려한 서비스보다, **운영 가능한 내부 도구**다.

---

## 4. MVP에서 반드시 증명해야 하는 것
MVP는 아래를 실제로 작동하게 만들어야 한다.

### 4.1 서사를 구조화할 수 있는가
최소 단위:
- Narrative
- Claim
- Evidence
- Counterpoint
- Actor
- Source
- State

즉, 기사나 리포트를 단순 저장하는 것이 아니라,
**어떤 주장이 어떤 서사의 일부인지**를 구조화할 수 있어야 한다.

### 4.2 지지와 반박을 동시에 다룰 수 있는가
각 서사에 대해 다음이 함께 있어야 한다.
- 왜 이 서사가 떠오르는가
- 어떤 근거가 지지하는가
- 어떤 반론이 존재하는가
- 그래서 현재 판단은 무엇인가

### 4.3 상태 변화를 관리할 수 있는가
각 서사는 최소한 아래 상태 중 하나를 가져야 한다.
- Emerging
- Growing
- Contested
- Stabilizing
- Fading

즉 서사는 정적 메모가 아니라 **시간에 따라 관리되는 단위**여야 한다.

### 4.4 제한된 소스로도 의미 있는 해석이 나오는가
처음부터 세상의 모든 소스를 다루지 않는다.

MVP는 오히려:
- 핵심 소스 20~30개
- 핵심 서사 5~10개
정도만으로도 구조화가 유효한지 확인해야 한다.

### 4.5 최소 개인화가 실제 인사이트를 주는가
개인화는 완전체가 아니라 아래 정도만 검증한다.
- source on/off
- source weight 3단계
- 메인 그래프와 개인 그래프 차이 확인

즉 개인화는 추천 시스템이 아니라,
**관점 차이 인식 도구**로 먼저 검증한다.

## 5. MVP에서 만들 것

### 5.1 내부 기능 (우선순위 최상)
#### Source 관리
- 소스 등록
- 소스 수정
- 소스 상태 분류
- 기본 가중치 설정

#### Narrative 관리
- 서사 생성
- 서사 수정
- 서사 상태 지정
- 상위/하위 서사 연결

#### Claim 관리
- claim 생성
- claim 수정
- claim을 narrative에 연결
- claim의 발화 주체(actor) 연결

#### Evidence / Counterpoint 관리
- evidence 추가
- evidence를 claim에 연결
- counterpoint 추가
- 반대 근거와 한계 정리

#### Actor / Entity 정규화
- 인물/기관/기업 중복 정리
- 동일 entity 통합

#### 최소 반자동 보조
- URL 입력
- 본문 추출
- claim 후보 제안
- keyword 후보 제안
- counterpoint 후보 초안

원칙:
- 자동 생성은 후보까지만
- 최종 반영은 사람이 승인

### 5.2 외부 기능 (얇은 레이어)
#### 메인 홈
- 서사 리스트
- 상태 배지
- 변화량 표시
- 핵심 키워드
- 간단한 sparkline 또는 시계열 단서

#### 서사 상세
- 한 줄 핵심 요약
- 지지 키워드
- 반박 키워드
- 현재 판단
- 관련 근거
- 관련 actor/source

#### 최소 개인화
- 소스 on/off
- 소스 가중치 3단계
- 메인과 개인 차이 간단 표시

## 6. MVP에서 의도적으로 만들지 않을 것
초기부터 욕심내면 망가질 가능성이 높은 것들은 제외한다.

### 제외 항목
- prediction market
- 탈중앙화 거버넌스
- 완전 자동 서사 추출
- 실시간 전체 업데이트
- 고급 graph visualization
- 분기/합류가 완전히 표현된 streamgraph 완성형
- 복잡한 추천 시스템
- 영상 콘텐츠 자동 생성
- graph DB 중심 구조

즉 MVP는 “최종 시스템의 축소판”이 아니라,
**최종 시스템의 핵심 가설만 검증하는 최소 구조**여야 한다.

---

## 7. MVP 데이터 범위

### 도메인
- 초기에는 **크립토**로 제한

### 소스 수
- 핵심 소스 20~30개

### 서사 수
- 핵심 서사 5~10개

### 데이터 층 우선순위
전부 다루지 않고 아래 순서로 좁게 시작한다.

1. 정치/거시경제 상위 방향층 일부
2. 메이저 VC/재단/파운더/거래소 등 산업 내부 핵심 집단
3. 리서치/해석 proxy 층 일부
4. 대중 뉴스/커뮤니티는 보조층

즉 **작은 but 고신호 데이터셋**으로 시작한다.

## 8. MVP 기술 스택

### 핵심 원칙
기술 스택은 가장 멋진 구조가 아니라,
**운영 가능한 서사 편집 시스템을 빨리 만드는 쪽**으로 고른다.

### 백엔드
**Python + FastAPI**

이유:
- 데이터 처리와 LLM 보조 추출에 강함
- 배치 작업과 궁합이 좋음
- admin/backoffice API 작성이 편함

### 프론트엔드
**Next.js**

이유:
- 내부 CMS와 외부 뷰를 함께 만들기 좋음
- 로그인/개인화 확장 여지가 있음
- UI 반복 속도가 빠름

### 데이터베이스
**Postgres**

이유:
- 초기 관계형 구조를 명확하게 잡기 좋음
- graph DB보다 운영이 단순함
- entity/relationship 관리에 충분함

### 인증
초기 우선순위:
- 관리자 로그인 먼저
- 일반 사용자 로그인과 개인화는 그 다음

### 본문 추출 / 자료 입력
초기 방식:
- URL 입력
- PDF/리포트 텍스트 추출
- 수동 붙여넣기 허용

즉 완전 자동 scraping보다,
**입력 안정성**을 우선한다.

### LLM 사용 위치
LLM은 아래에만 사용한다.
- claim 후보 추출
- summary 후보 생성
- keyword 후보 생성
- counterpoint 후보 초안
- source 분류 보조

원칙:
**LLM은 편집 보조자이지 최종 판정자가 아니다.**

### 시각화
MVP에서는 단순하게 시작한다.
- list
- badge
- delta
- sparkline

stream/river 계열은 후속 버전에서 고도화한다.

## 9. MVP 개발 순서

### Phase 1. 데이터 모델 + admin backend
- Postgres schema
- CRUD API
- Source / Narrative / Claim / Evidence / Counterpoint / Actor 관리

### Phase 2. 관리자 CMS
- source 등록
- narrative 연결
- claim 수정
- counterpoint 정리
- 상태값 조정

### Phase 3. 외부 읽기 화면
- 메인 홈
- 서사 상세
- 변화량 표시

### Phase 4. 반자동 추출 보조
- URL 입력 → 본문 추출
- claim 후보 생성
- keyword 후보 생성
- 사람이 승인

### Phase 5. 개인화 최소 기능
- 소스 on/off
- 소스 weight 3단계
- 메인 vs 개인 차이

## 10. MVP 성공 기준
MVP는 아래가 되면 성공이다.

- 제한된 소스로도 서사를 구조화할 수 있다.
- claim / evidence / counterpoint / state 관리가 실제로 돌아간다.
- 사용자가 기사 하나를 봤을 때, 기존 뉴스 소비보다 더 나은 맥락 해석을 얻는다.
- 내부 편집자가 서사 품질을 유지할 수 있다.
- 메인과 개인 그래프 차이가 의미 있는 인사이트를 만든다.

즉 성공 기준은 “예쁜 그래프”가 아니라,
**운영 가능한 서사 편집 시스템이 실제 가치를 주는가**다.

---

## 11. 원래 목표와의 관계
이 MVP 1차안은 최종 비전의 축소판이 아니다.

오히려:
- 최종 비전의 핵심 가설을 검증하기 위한
- 현실적인 시작점이며
- 이후 확장 가능한 편집 기반을 만드는 단계다.

정리하면:
- **원래 목표**: 서사 기반 컨텍스트 인텔리전스 서비스
- **MVP 1차안**: 그 목표를 검증하기 위한 반자동 서사 편집 CMS

즉,
**원래 이런 걸 시도하는 것이 목적이지만,
MVP에서는 먼저 이렇게 만든다**
가 이 문서의 핵심이다.
